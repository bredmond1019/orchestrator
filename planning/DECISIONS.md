# DECISIONS — Settled Choices & Their Reasoning

*Append-only. Records non-obvious decisions so they don't get relitigated. Newest at the bottom.*
*This is the "why" that outlives `STATUS.md`'s deviation log. If a choice here is reversed, add a new entry superseding it — don't edit the old one.*

Format per entry: **what was decided · why · what was rejected and why.**

---

### D1 — Expertise first; business/job follows
**Decided:** The goal is expertise in AI/agentic/harness engineering. A studio, a job, or a product are outcomes that follow, and the plan is built so identical work serves all three.
**Why:** Removes the false choice between "build" and "earn," and resolves the build-vs-sell tension — building the full project set *is* the goal, not a detour from revenue.
**Rejected:** The original "build, ship, get paid in 16 weeks" framing — too revenue-pegged for someone whose real driver is mastery and who has runway.

### D2 — Sequence, not calendar
**Decided:** Plans are ordered by dependency and competence, with no dates assigned. Cadence comes from Brandon's separate whiteboard system.
**Why:** "Let it breathe" — no rush, life events (job hunt, family) make dates brittle. Dependencies are real; timing is not.
**Rejected:** The original week-by-week 16-week schedule — broke on first slip and induced guilt.

### D3 — Drop the Socratic Tutor as the organizing goal
**Decided:** Projects are sequenced by "does this teach something sellable / make me demonstrably expert," not by assembling one app. The Socratic Tutor and physics study are archived in separate docs and excluded from this set.
**Why:** The tutor's dependency graph was distorting the project order; the real goal is general expertise, and the tutor is a crowded, hard-to-defend market (researched).
**Rejected:** Keeping the tutor as the capstone — it pulled the timeline away from expertise/revenue and isn't defensible as a startup as originally conceived.

### D4 — Project G (agent memory) is the centerpiece, retained at full weight
**Decided:** The episodic→semantic memory system stays as the differentiating build.
**Why:** Durable agent memory with confidence decay and contradiction handling is frontier-adjacent, the thing most teams get wrong, and underpins two of the three product ideas. Given expertise-first, it's the capstone.
**Rejected:** Cutting/deferring it as "premature" — valid under revenue-first, wrong under expertise-first.

### D5 — Testing scope: Option A (core only)
**Decided:** Test the core engine, infrastructure, and services; fix four documented production bugs; do **not** test the reference-only customer-care workflow. Then every new workflow ships with its own tests.
**Why:** Customer-care is disposable reference code Brandon won't extend; testing it spends effort on throwaway. The same testing patterns are learned by testing code that's kept.
**Rejected:** Option B (full sweep including customer-care) — more thorough but wastes time on code that won't ship.

### D6 — Python for orchestration; Rust only where it genuinely wins
**Decided:** The orchestration framework stays Python (I/O-bound — model/DB/network latency dominates). Rust is reserved for genuinely CPU/latency/memory-bound work.
**Why:** Rewriting orchestration in Rust is a *learning* win, not an *architecture* win; microseconds saved are meaningless behind a 2-second model call.
**Rejected:** Porting the engine to Rust for performance — already done once as learning; not worth maintaining as the production path.

### D7 — Rust's home: the CLI now, an inference runtime only later (with data)
**Decided:** The Rust Harness CLI is the near-term Rust project (instant startup, single binary, daily use — keeps the skill warm through genuine use). A Rust local-inference *runtime* layer is kept in mind but **not** in scope; it gets built only if a measured limit makes off-the-shelf serving (Ollama et al.) insufficient.
**Why:** The CLI's Rust advantages are unambiguous and immediate; the runtime's are situational and announce themselves with data. The MIDI tool was dropped — it was an excuse to write Rust, not a real need.
**Rejected:** Forcing a Rust project for its own sake (MIDI tool); pre-building the runtime before a bottleneck is measured.

### D8 — Project H (model eval) is offline evaluation, NOT a runtime router
**Decided:** The eval harness runs occasionally to *produce* per-node routing decisions that bake into each node's `model_provider` at design time. It does not select models per-request at runtime.
**Why:** Static per-node decisions capture most of the value; per-request runtime selection adds latency and complexity for marginal benefit. The expert skill is the measured routing *judgment*, not dynamic switching.
**Rejected:** A runtime model router — overkill; the impressive-but-unjustified trap.

### D9 — Existing production work is foregrounded as case studies
**Decided:** The Internal Support Dashboard (100+ users, 24–48hr wait-time cut, solo, still in daily use), Helpscout automation (solo, production RAG), and AI Scribe (heavy contribution, honest framing — not architect) are featured as proof of existing expertise. Project D is framed as reinforcing the proven Helpscout pattern, not first contact.
**Why:** Brandon undersells; these are stronger than anything not-yet-built and reposition him from "aspiring" to "proven."
**Rejected:** Treating the portfolio as only forward-looking — a material undersell.

### D10 — Public narrative: subject-is-always-you; never name the company
**Decided:** In anything public, the subject of every sentence is Brandon, his work, or his reasons — never the previous company's conduct. The company is never named in posts (LinkedIn carries the factual record).
**Why:** A non-disparagement agreement was signed. The asymmetry (talk about yourself, not them) is both legally safe and more dignified. Two suspected-"slop" repos are de-featured but repurposed as an honest "early adopter, learned volume isn't quality" blog post rather than deleted.
**Rejected:** Any framing that describes the environment ("toxic," "difficult") — that's a claim about them, however softened.

### D11 — Documentation: separate orientation from state; minimum-context by default
**Decided:** Five planning files with distinct jobs — CONTEXT (orientation/router, stable), STATUS (state, volatile), the three plans (content), plus README (navigation) and this DECISIONS file. Pass only the subset a question needs. CONTEXT routes and never duplicates the plans; STATUS mirrors only names, never content.
**Why:** Orientation and state age differently; folding them together makes the whole thing feel stale on every task completion. Single-source-of-truth prevents drift.
**Rejected:** One fat CONTEXT.md containing everything — goes stale fast, forces passing bloat for narrow queries.

### D12 — Repo strategy: one Python monorepo; separate repos only for different languages/deployables
**Decided:** The Python framework and all Python workflows (Projects A–H) live in **one monorepo** — every project is a workflow directory (workflow + nodes + prompts + tests) added alongside the existing ones, not a clone. Separate repos only for genuinely different languages/deploy lifecycles: the Rust CLI, the website, a future Rust runtime.
**Why:** The architecture is "framework as scaffold, projects as attached workflows," with heavy verbatim component reuse across projects. Clone-per-project would fragment the framework into divergent copies with no clean way to propagate fixes. Test for "new repo?": *does it share the Python framework's code/dependency tree?* If yes, same repo.
**Rejected:** Cloning the orchestration repo per project — optimizes for unwanted isolation at the cost of the reuse that is the whole design.

### D13 — Per-repo agent context (CLAUDE.md) and daily log (DEVLOG.md); just-in-time task specs
**Decided:** Each code repo carries its own `CLAUDE.md` (how an agent works in that repo — conventions, build/test commands, the tests-ship-with-every-workflow rule) and `DEVLOG.md` (append-only daily working log, repo-scoped). Per-block work orders are generated **just-in-time** into a `tasks/` folder when a block starts, each with explicit **acceptance criteria** — never pre-written for all blocks.
**Why:** Distinct jobs, no overlap: planning docs describe the endeavor; CLAUDE.md describes working in a repo; DEVLOG records repo history; STATUS rolls up cross-repo state; DECISIONS records why. Generating task specs just-in-time avoids planning-mode procrastination while a fixed template prevents rethinking the convention each time.
**Rejected:** Pre-writing all task specs up front (procrastination trap); a single global devlog (loses repo-local detail); reusing the planning CONTEXT.md as repo agent-context (different reader, different job — hence separate CLAUDE.md per repo).

### D14 — The destination has a named product: the Company Brain
**Decided:** The three RFS ideas Brandon was drawn to (YC S26: Company Brain, AI Operating System for Companies, Software for Agents) are not three options to choose between — they are three layers of **one** system the plan now names explicitly: a **Company Brain** that ingests a company's scattered knowledge, structures it, keeps it current via the Project G memory engine, and emits an executable skills file agents can act on. This is the product the existing project library was already converging toward (D + G + product idea #1), now stated as the destination rather than left implicit.
**Why:** Founder-market fit is unusually strong and *lived*, not hypothetical — Brandon watched a previous employer grow 30→200 people, diagnosed the exact knowledge-fragmentation failure Company Brain describes, proposed this literal solution internally, and was turned down, then watched the cost compound. That is conviction grounded in firsthand pain, the strongest signal available. Company Brain is also the best *architecture* match to what's already built: Project D is the ingestion/retrieval layer (already shipped in production as Helpscout), Project G is the "keeps it current" engine (decay + contradiction handling), and the skills-file output is the Claude Code skills primitive the harness already runs.
**Rejected:** Treating the three RFS ideas as a menu and picking one; and the prior "endgame is open / three open doors" framing as the *primary* stance — a destination is now named because decisions prune faster against one. (A job remains a fine outcome and identical work still serves it — D1 stands — but the build now aims at a specific product.)

### D15 — Buyer wedge: fast-growing SMBs caught at the 30–80 inflection, privacy-first
**Decided:** The primary wedge is fast-growing small companies at roughly the 30–80-employee inflection point — past the size where tribal knowledge fits in a few heads, before the size where a Notion/wiki graveyard has already been built and abandoned. Positioning is privacy-first: the company's knowledge **never leaves the building**. Enterprise is an explicitly welcomed later expansion (Brandon is open to it), served by the same core (see D16), not a different product.
**Why:** This resolves the earlier "a 12-person gym has nothing to extract" objection — the wedge is an *inflection*, not a size. It matches Brandon's lived experience exactly (the 30→200 company). Instant-value-then-compounding fits a pre-pain buyer: Project D answers from existing docs on day one, Project G makes it smarter over time. The privacy angle is a wedge funded competitors structurally *won't* take, because cloud multi-tenant SaaS is correct for *them* (it's how you scale to a thousand customers) — leaving on-prem/private uncontested. In Brazil specifically, LGPD plus wariness of US cloud makes "never leaves the building" concrete, not abstract.
**Rejected:** Pure-SMB-only (forecloses the enterprise upside Brandon wants open); enterprise-first (most crowded, best-capitalized lane on the RFS page, worst fit for a solo builder, weakest use of the privacy edge); and "you'll need this later" as the pitch (the buyer doesn't feel the pain until too late — the pitch must deliver day-one value).

### D16 — Architecture: one deployment-agnostic Python brain, two shells (Rust SMB appliance + cloud enterprise)
**Decided:** Build the Company Brain as **one** Python core (the existing orchestration framework + Projects D, G, H + skills-file emitter) that is **deployment-agnostic** and exposes a single clean HTTP API. Two *shells* wrap the same core: (1) an **SMB shell** = a **Rust single binary** that supervises a local Python brain + local Postgres/pgvector + local models, owns first-run setup, is the operator's local interface (clap CLI, optionally ratatui TUI), and surfaces Project H cost/quality numbers; (2) an **Enterprise shell** = a thin cloud control plane (multi-tenant, SSO, audit, admin dashboard, integration connectors) built **only when an enterprise pulls**. The brain is written **once**; shells are new surface around it, never a rewrite.
**Why:** Avoids the trap of building the appliance, getting enterprise interest, and discovering a rewrite is needed. SMB and enterprise stop being two products and become one core with two faces. Maps the three RFS ideas onto layers cleanly: **Company Brain = the brain (Layer 1); Software for Agents = the API/agent interface (Layer 3); AI OS for Companies = the enterprise shell (Layer 2).** Brandon is already building the hardest layer (the brain).
**Rejected:** Two parallel codebases for SMB and enterprise (doubles maintenance, guarantees drift); building the enterprise shell speculatively before a real enterprise pull (premature, and the AI-OS lane is brutally competitive); putting deployment logic inside the brain (see D18).

### D17 — Rust earns its place as the appliance shell, not as orchestration
**Decided:** Rust's role is upgraded from "personal ops CLI that keeps the skill warm" (D7) to **the SMB appliance shell** — the single static binary that is itself the product's privacy promise ("copy one file, double-click, your data stays in the building"). This *supersedes the scope* of D7's CLI (the CLI work becomes the appliance shell) while *preserving* D6 and D7's core ruling: orchestration stays Python, and a Rust inference *runtime* is still out of scope until measured data demands it.
**Why:** The single-binary, instant-startup, no-dependency-hell properties of Rust are not a nice-to-have here — they are the *only* credible delivery vehicle for "runs on your hardware, a non-technical operator can run it, nothing leaves the building." A Python app cannot make that promise convincingly (environment, Docker, dependency hell for the client). This is exactly the "where Rust compounds" the plan already identified (single-binary appliances for non-technical operators) — now pointed at a product, not just daily ops. Clean language boundary preserved: **Rust commands and observes, Python executes**, over HTTP — no FFI, no rewriting working Python.
**Rejected:** Keeping Rust as a dev-only convenience tool (undersells where it actually wins); rewriting any brain logic in Rust (violates D6); the Rust inference runtime now (still announces itself with data per D7).

### D18 — The one discipline that keeps it one product: no deployment logic in the brain
**Decided:** Two things are **injected, never hardcoded** into the Python brain: (1) **where models run** — per-node `model_provider` comes from a config file that is the *output of Project H* (this node is safe on local-70B; that node needs Claude); SMB config points mostly at local Ollama, enterprise config at a private/cloud endpoint; (2) **where data lives** — all persistence goes through the existing `GenericRepository` against Postgres+pgvector, local in SMB, managed/multi-tenant in enterprise. **Rule:** the first `if running_locally:` written inside a brain node means two products have started being built. Deployment decisions live in config and in the shell, never in the brain.
**Why:** This single discipline is what makes D16's "write the brain once" actually hold in practice. Both injection points already exist in the codebase (`model_provider` is config; persistence already goes through `GenericRepository`), so this is enforcing a boundary that's already there, not adding machinery.
**Rejected:** Branching on environment inside nodes (the concrete mechanism by which one product silently becomes two).

### D19 — The privacy wedge is real *today*, with an honest qualifier; "local-by-default, frontier-for-the-few"
**Decided:** The local-model privacy wedge is treated as viable **now** (mid-2026), not 12–18 months out as the Master Plan previously hedged — but with two honest qualifiers baked into the pitch. (1) **Hardware honesty:** the pitch is *not* "runs on whatever you have." A trustworthy Company Brain needs a 70B-class model at Q4, which wants a 64–128GB unified-memory machine or a single 24–48GB-VRAM GPU — a low-thousands capital cost, *cheaper and private*, not *worse but private*. (2) **Routing honesty:** the hardest Project G consolidation step (messy episodes → durable, falsifiable facts with correct confidence) is the one task where a weak model silently corrupts everything downstream and must stay on Claude. So the promise is **"local-by-default, frontier-for-the-named-few-steps,"** with Project H producing the auditable list of which steps go where and what each costs — shown to the operator in plain numbers.
**Why:** Research (June 2026) shows open-weight models closed the gap hard; for RAG specifically, retrieval quality matters more than raw model size, and a good embedder + clean chunking + a smaller local LLM beats a giant model with poor retrieval — and a company brain is mostly retrieval + structured extraction, which is exactly what local models do well. Absolutism ("nothing ever leaves") would be a lie given the consolidation-step reality; the honest, audited version is *more* defensible because the operator sees the tradeoff. This updates the Master Plan's "the bet may be a year ahead of the pain" hedge: the *technology* is ready now; the funding discipline (services revenue funds the bet) still stands because the *buyer's felt pain* may still lag.
**Rejected:** "Runs on any hardware you already own" (false for the office-desktop tier — 3–8B models can't carry a trustworthy brain); absolute "nothing ever leaves the building" (false given the consolidation step; an asterisk that would eat the pitch if discovered rather than disclosed); and continuing to treat the local bet as premature (the tech caught up).

### D20 — Self-improvement boundary: agents evolve what's gated; new capability enters by PR; the gates are never self-approved
**Decided:** A clear, permanent boundary governs every "self-evolving / self-building" capability (now and future). The system **may freely do** the things that are *measured and gated*: self-correct outputs (critic→revise + outcome signals), evolve **prompts** (candidate variants scored by Project H, winner promoted), evolve **routing** (H re-run on accumulated real examples, config updated — still offline, never per-request, per D8), evolve **memory** (Project G consolidation already does this), and **compose new workflows over the existing library of trusted nodes** (agent emits a `WorkflowSchema`; the `WorkflowValidator` rejects illegal DAGs *before* run; candidates run against test inputs in a sandbox before registration). The system **may not autonomously author and deploy** the things those gates depend on. **New capability (new node code, e.g. a `NotionNode` wrapping an MCP server) enters only through a human-reviewed pull request:** an agent researches, writes the node conforming to the `Node` ABC, writes its `.j2` prompt and — non-negotiable per the tests-ship-with-every-workflow rule — its tests, and opens a PR; CI runs the tests, the validator checks example workflows, and **a human reviews and merges.** Review happens **once per capability, not once per use** — once merged, any agent composes the node freely. **The PR gate, the validator, the test-runner, the capability registry's notion of "trusted," the eval rubric, and the Project G consolidation prompt are human-owned infrastructure that agents may propose changes to but never self-approve.** The thing being graded never rewrites its grader.
**Why:** This is the seam between "every agent action needs approval" (unusable) and "no agent action needs approval" (unsafe) — the same seam Git/GitHub already found: trusted committers, reviewed contributions, free use of what's merged. It makes self-construction safe *without inventing a new safety mechanism* — it reuses the one the industry hardened over twenty years (the PR). It directly extends the Software-for-Agents thesis (Layer 3, D16): agents don't just *use* the framework, they *extend the toolset* through a disciplined contribution process — but capability expansion stays slow and human-gated while composition stays fast and free. It preserves the project's core anti-pattern defense (silent compounding failure): an uncaged self-written node poisons every workflow that composes it, and a self-modified consolidation prompt corrupts the memory everything trusts — both prevented by keeping authorship-of-new-capability behind the PR and keeping the gates out of agent reach.
**Rejected:** Agents authoring and deploying new node code without human review (reintroduces silent compounding failure at the worst layer); per-use approval of agent-composed workflows (unusable, and unnecessary once components are vetted); letting agents self-approve PRs or modify the validator/rubric/consolidation-prompt autonomously (the grader-rewrites-itself failure); and **building the agent-contribution *platform* before hand-building nodes is a felt, repeated chore** — the infinite-internal-tooling trap wearing a more seductive costume (see Agentic plan, Part 5: the trigger is real repeated friction, not "it'd be cool"). This is a Phase 3+ capstone that *assembles* from the plan (clean API + node library + validator/test-gate + Project H), not a detour from it.

### D21 — Project A is a personal knowledge feed first, blog engine second; digest-always, blog-on-flag; static HTML on the Mini
**Decided:** Project A is extended from "YouTube → blog" into a **dual-input, dual-output personal knowledge pipeline.** Inputs: a YouTube URL (transcript flow, existing) **or** an article URL (new `FetchArticleNode` + `SourceRouterNode`). Outputs: **every** item produces a categorized **personal digest** (the default path), and **only when an optional `make_blog` flag is set** does it also produce a public blog draft through the self-critic→revise loop. The reading surface is **static HTML served by Caddy on the Mac Mini, privately** — one page per item plus per-category index pages — readable on the Pixel tablet/phone browsers and the Kindle's browser. `SummaryOutput` gains `category` (small growable set: ai_engineering / physics_relativity / music / other), `tl_dr`, and `read_time_estimate`. Embeddings are stored at write time for *every* item.
**Why:** Makes Project A useful on Day 1 and *every morning* (Brandon constantly finds AI-engineering videos and physics/music articles he loses track of), which means the project gets lived-in rather than built-once. Digest-always/blog-on-flag keeps the default path simple and makes the blog branch opt-in, so the pipeline never has to *decide* blog-worthiness. Static HTML on the Mini beats Notion on three grounds: it **compounds into the harness/appliance work** Brandon is doing anyway (Caddy is already up in Foundation Block B), it's the **universal device format** (tablet/phone/Kindle, no app or login), and it **teaches transferable patterns** (self-hosting, static generation) rather than the Notion API. It's also consistent with the privacy-first ethos of the whole practice. Strategically: this personal feed **is a one-person Company Brain** — the dogfood version of the product — and `FetchArticleNode` is reused by the real Company Brain to ingest web-based client knowledge.
**Rejected:** Notion as the store (third-party dependency, poor Kindle story, teaches a vendor API not a transferable pattern, mild tension with privacy-first ethos); building both Notion and HTML to compare (wasteful — the reasoning already favors HTML); per-item or auto "is this blog-worthy?" decisioning (adds a decision the MVP doesn't need; the flag is simpler); a fixed/rigid category taxonomy (let it grow from a small starting set).

### D22 — Project A MVP boundary: ingestion + store + dumb display now; search and "what I know" via Projects F/G later
**Decided:** The Day-1 Project A ships the ingestion-and-store layer and a deliberately **dumb** reading surface only. It explicitly does **not** include: a tagging UI, a search box, "what I already know" intelligence, cross-device sync, recommendations, one-tap phone capture apps, or Kindle EPUB generation. Capture on Day 1 is just a POST to the existing endpoint (bookmarklet / phone share-shortcut / `send_event` script). Search attaches later via **Project F** (over the same embeddings); "what I already know" / dedup attaches later via **Project G** (memory over the same artifacts); one-tap capture and weekly Kindle EPUB are just-in-time upgrades built only when the MVP's friction is actually felt.
**Why:** The smart layers attach **for free later precisely because embeddings are stored at write time now** — so there is no reason to build them Day 1, and every reason not to. Building a personal-knowledge *platform* (reader UI, search, sync, recommendations) on Day 1 is the infinite-internal-tooling trap in its most seductive form (Brandon's documented weakness, here wearing a favorite-hobby costume). The discipline: the *pipeline* is the project; the reading surface is the thinnest thing that lets him read on his devices; upgrade the surface only when daily use demands it.
**Rejected:** Building search/tagging/recommendation now (duplicates Projects F/G, delays the rep that teaches the agentic pattern); a capture app on Day 1 (premature — a POST suffices until friction is felt); EPUB/Kindle generation now (someday item; static HTML covers the Day-1 morning-coffee case).

### D23 — Mac Mini two-face architecture: Caddy+Cloudflare for public, Tailscale for private; these are separate tools for separate jobs
**Decided:** The Mac Mini serves two entirely different audiences with two architecturally separate networking paths. **Public face** (Caddy + Cloudflare DNS + port 80/443 forwarding): hosts `learn-agentic-ai.com` and the public blog — accessible to anyone with the URL, protected by Cloudflare, SSL via Caddy/Let's Encrypt, home IP hidden. **Private face** (Tailscale): the personal knowledge feed, orchestration API, Celery dashboard, Postgres, Claude Code remote triggers, and all other private tooling — reachable only from devices on the tailnet (Brandon's Pixel tablet, Pixel phone, Kindle, laptop), completely off the public internet with no open ports. Tailscale's Personal plan (free forever) covers unlimited devices for a single user with MagicDNS, subnet routing, and WireGuard encryption. The two faces share a physical box but are logically and network-separated.
**Why:** Tailscale cannot make a public website accessible to strangers — it is a private mesh network by design, and that is its value. Attempting to serve `learn-agentic-ai.com` through Tailscale would make it unreachable to anyone who hasn't been added to the tailnet, defeating the purpose of a public portfolio. The two-face architecture gives each concern the right tool: public-facing traffic gets a hardened public reverse proxy (Caddy + Cloudflare), private tooling gets a zero-trust private network (Tailscale). The personal knowledge feed is *correctly* private — it's yours, Tailscale enforces that at the network level. Headscale (open-source drop-in for Tailscale's coordination server) is the escape hatch if the proprietary control plane ever becomes a concern, but is not needed now.
**Rejected:** Routing the public portfolio site through Tailscale (makes it private — defeats the purpose); using Tailscale Funnel for the public site (routes through Tailscale's infrastructure, adds uptime dependency, unnecessary when Caddy+Cloudflare already solves it correctly); a single unified public setup for everything (exposes private tooling to the internet).
**Amendment (2026-06-10) — public face implemented via Cloudflare Tunnel, not port-forwarded Caddy.** When the public face actually shipped, `learn-agentic-ai.com` was served through a **Cloudflare Tunnel** (`cloudflared` running on the Mini, outbound-only connection to Cloudflare's edge) rather than the originally-specified Caddy + Cloudflare DNS + port 80/443 forwarding. The decision and its *intent are unchanged* — two architecturally separate faces, public traffic hardened by Cloudflare, home IP hidden — but the tunnel is a strictly better fit for a privacy-first harness: it opens **zero inbound ports** on the Mini (no router port-forwarding, no exposed home IP, no inbound attack surface), where the original Caddy path required forwarding 80/443. Cloudflare still fronts DNS/TLS/DDoS; the origin reaches out to Cloudflare instead of Cloudflare reaching in. Note this is **not** Tailscale Funnel (still rejected above) — it is Cloudflare's own tunnel, which keeps the public face entirely on the Cloudflare path and off the tailnet. The private face is unchanged: Tailscale, no open ports. Net effect: both faces of the Mini now have **no inbound ports open at all**.

### D24 — Firecrawl role: trafilatura-first for single articles, Firecrawl-fallback for JS/paywall, CrawlSiteNode for Company Brain site ingestion; free tier until a real crawl demands upgrade
**Decided:** Firecrawl is adopted with a **defined role alongside trafilatura**, not as a replacement. `ArticleExtractionService` uses **trafilatura as the default** (free, local, no credits, fast for clean articles) and **Firecrawl as a fallback** for JS-rendered pages or cases where trafilatura returns empty/junk. Separately, Firecrawl's `/crawl` endpoint powers a **`CrawlSiteNode`** for the Company Brain's site-ingestion path (entire client help-doc sites or wikis by URL). Firecrawl's MCP server support enables future agent tool use without extra wiring. Start on the **free tier (500 credits/month)**; upgrade only when a real client crawl demands it. **When Firecrawl runs inside an agent tool loop, add a `max_calls` guard** — same discipline as Project B's `max_iterations`.
**Why:** trafilatura handles the vast majority of personal feed extraction without cost or external dependency. Firecrawl earns its place for three specific cases no free local library handles: JS-rendered pages, systematic multi-page site crawling (the Company Brain client ingestion path), and MCP-native agent tool access. These are real gaps; using it for everything adds per-credit cost where a free library suffices.
**Rejected:** Replacing trafilatura entirely with Firecrawl (unnecessary cost and dependency for simple articles); self-hosting Firecrawl (AGPL, extra infra, misses managed features); using only trafilatura (can't handle JS-heavy pages or site crawls); skipping the `max_calls` guard in agent contexts (silent credit burn).

### D25 — Honcho as Project G reference architecture; personal feed as Honcho validation experiment; build your own G for the Company Brain
**Decided:** Three related choices. **(1) Honcho as reference architecture for Project G:** before writing any Project G code, read Honcho's source (github.com/plastic-labs/honcho) and internalize its design. Adopt directly: the two-stage pipeline (ingest-time fast extraction with a small model + dream-time background consolidation with a stronger model), the multi-peer entity model (evolve to a `Peer`-centric schema supporting company/client/product/process entities and "what does A know about B" queries), and the natural-language query interface (memory loader takes a question string, returns a synthesized answer, not just top-k cosine chunks). **(2) Personal knowledge feed as Honcho validation experiment:** when the Phase 3 memory upgrade arrives for the personal feed, implement it using Honcho running on the Mini (same Postgres + pgvector + Redis stack, Docker Compose, pointed at Claude Haiku) rather than the custom G. Live inside it daily with deliberate observation: where is it strong, where is it awkward, where does the domain mismatch (conversational preferences vs. organizational knowledge) show up in practice. This is competitive intelligence felt from the inside, not read in docs. **(3) Build your own G for the Company Brain in production:** not Honcho. Domain specificity (Honcho's ingest-time model was trained on conversational preference data; Company Brain needs organizational knowledge — SOPs, client relationships, product facts, evolving processes), privacy-first deployment (Honcho managed sends data to their API; yours runs on the client's hardware), full-stack traceability (every extraction step debuggable and improvable via `.j2` prompts and Part 5's evolution loop), and differentiation (can't be asked "why not just use Honcho?" if the answer is "built specifically for organizational knowledge, runs on your hardware, every step is visible").
**Why:** Honcho is the best available open-source reference (90.4% LongMem S, 5% median context, two-stage pipeline, peer-centric model — all validated by published benchmarks). Using it as a reference and a personal-feed competitor evaluation is the highest-ROI approach to Project G. Using it in the Company Brain product would trade the most defensible differentiator for convenience.
